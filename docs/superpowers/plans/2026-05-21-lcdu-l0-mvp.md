# LCDU L0 MVP 实施设计

**日期：** 2026-05-21

**适用范围：** `cc-社会计算器` Research 线，policy-reaction 自动化更新方法在 Route-B 止损后的下一阶段切换。

**当前状态：** L0 MVP 设计稿。

---

## 1. 目标

LCDU 的目标不是继续搜索更好的 patch，而是验证：

> 把更新对象从 prompt/patch 提升到“latent state + 分布约束程序”之后，是否能形成一个更合理、可审计、可迁移到 Product 的候选表示层。

L0 只回答两个问题：

1. 这种表示能不能稳定编译成 Product 可消费候选；
2. 这种表示能不能形成一组边界清晰、可进入后续 runtime gate 的候选 artifact。

---

## 2. 为什么是 LCDU

S2PC 和 Route-B 现在已经共同暴露了一个问题：

- 能找到局部 signal；
- 但 signal 主要出现在 patch / 机制配置层；
- 一旦扩到 repeat，稳定性不足。

这说明下一步不该优先换搜索器，而该换更新表示。

LCDU 的核心假设是：

> policy-reaction 的校准误差更适合表示成“latent state 偏移”和“群体分布约束偏离”，而不是直接表示成 persona prompt patch。

---

## 3. L0 范围

### 3.1 本轮要做的

1. 定义最小 latent state 集合；
2. 定义最小群体约束模板；
3. 实现 `latent + constraint -> candidate_prompt_components` 编译器；
4. 生成一组 strict JSON 候选 artifact；
5. 保持与现有 Product runtime 输入兼容。

### 3.2 本轮不做的

1. 不直接跑 runtime cohort；
2. 不新增 Product 代码路径；
3. 不做 repeat-aware gate；
4. 不引入新 provider；
5. 不做自由文本 prompt rewrite。

---

## 4. 最小 latent state 集合

L0 只保留三个 latent：

1. `institutional_trust`
2. `household_cost_pressure`
3. `policy_relief_urgency`

选择原因：

1. 与现有 `trust_only` 证据链有连续性；
2. 能覆盖“信任、成本压力、政策救助需求”三个最基本方向；
3. 足够小，便于后续止损。

---

## 5. 最小约束模板

L0 只保留四类约束：

1. `primary_policy_floor`
   - 主政策支持率不能低于某个保守下界

2. `primary_vs_secondary_gap`
   - 主政策与次政策之间的差值必须落在一个区间内

3. `distribution_smoothness`
   - 不允许把三政策分布压成过强尖峰

4. `segment_rank_guard`
   - 不允许破坏当前 segment 的主要排序逻辑

这些约束不是最终评估器，而是候选构造程序的一部分。

---

## 6. 编译策略

LCDU L0 不直接输出抽象 latent，而是编译成两层：

1. **结构化候选元数据**
   - `latent_state_updates`
   - `constraint_program`

2. **Product 可消费候选**
   - `candidate_prompt_components`
   - `best_candidate.parameter_patches`

编译原则：

1. `latent_state_updates` 只作为 Research 解释层；
2. Product 仍然只消费现有候选 schema；
3. 编译结果必须保留 strict JSON；
4. 不允许引入新的 Product schema 依赖。

---

## 7. 候选集合

L0 候选先固定成 4 个，不做搜索：

1. `l01`
   - `trust_up + urgency_up + smooth_distribution`

2. `l02`
   - `cost_pressure_up + urgency_up + paired_balance`

3. `l03`
   - `trust_up + cost_pressure_down + guarded_primary_gap`

4. `l04`
   - `trust_down + uncertainty_retain + distribution_guard`

设计目的：

1. 保留两类正向探索项；
2. 保留一类混合约束项；
3. 保留一类保守反向对照项。

---

## 8. 验收标准

L0 成功只需要满足以下条件：

1. 候选生成脚本通过测试；
2. 候选 artifact 严格符合 JSON；
3. 每个候选都能导出为现有 Product 可消费 schema；
4. 候选显式记录 latent 和 constraint 信息；
5. 不修改 Product runtime。

如果满足这些条件，就说明：

> LCDU 至少在“表示层”和“证据链层”是成立的，值得进入下一步 runtime gate。

---

## 9. 止损条件

L0 也有明确止损边界：

1. 如果连候选编译都需要改 Product schema，停止；
2. 如果 latent 表示最终还是退化成当前 patch 文案的另一种包装，停止；
3. 如果候选不能给出比 Route-B 更清晰的解释边界，停止。

---

## 10. 下一步

L0 完成后，下一步才进入：

1. `12x3 seed11` 单轴 runtime 预筛；
2. 与 baseline / `s02` 单点结果比较；
3. 只有当候选不弱于当前最强单点时，才进 repeat。
