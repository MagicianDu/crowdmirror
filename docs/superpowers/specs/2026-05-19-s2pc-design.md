# S2PC 语义-结构化人格校准算法设计文档

**日期：** 2026-05-19

**适用范围：** `cc-社会计算器` Research 线，后续通过 accepted artifact 转移到 Product 线。

**当前状态：** 设计评审稿，不包含代码实现。

---

## 1. 背景

当前项目已经形成了 policy-reaction 方向的可审计证据链：

- Product 侧可以运行 LLM cohort，并导出 segment-level prediction artifact。
- Research 侧可以用 HTOPS/HPS calibration split 生成校准输入，用 held-out split 做验收。
- `policy-reaction-update-method-gate-current-001` 已经能把不同候选更新方法分为 `accepted`、`rejected`、`diagnostic_only`。
- 当前最稳定有效的方法是 `calibration_split_prompting`。它在 gpt-oss 和 DeepSeek v4-flash 上都有 accepted 记录。
- 当前 `structured_persona_parameter_patch` runtime stability 是 rejected。
- 当前 DeepSeek v4-pro TextGrad 只有 W3/W4 诊断证据，没有 policy-reaction held-out evidence，因此只能是 diagnostic-only。

这说明项目的关键缺口不是“能不能让 LLM 写出新的 prompt”，而是：

1. 如何从校准误差中自动发现可解释的人群行为因素；
2. 如何把这些语义因素变成可搜索、可复现、可验收的 persona/prompt 参数；
3. 如何避免自由文本 prompt rewrite 带来的不可控更新；
4. 如何让 Research 方法直接增强 Product 的技术壁垒和客户说服力。

S2PC 的目标就是补上这个缺口。

---

## 2. 算法定义

**S2PC：Semantic-Structured Persona Calibration**

中文名：**语义-结构化人格校准算法**

核心思想：

> 语义检索负责发现“该调什么行为维度”，结构化参数搜索负责决定“调多少、是否接受”，held-out gate 负责决定“能不能进入证据链”。

S2PC 不是 TextGrad 的改名版本。它不让模型直接重写整段 prompt，而是把更新拆成三层：

- **语义层：** 哪些社会经济或心理因素解释了残差；
- **参数层：** 哪些 persona/prompt 参数需要调整，调整范围是多少；
- **证据层：** 调整后的候选是否在 held-out benchmark 上降低 loss。

---

## 3. 研究目标

S2PC 的 Research 目标是形成一个可作为论文贡献的自动校准方法：

1. **自动化：** 从 calibration residual 自动生成候选，不依赖人工逐条改 prompt。
2. **可解释：** 每个参数更新都能追溯到 segment、policy、residual 和 semantic factor。
3. **可复现：** 候选搜索空间、参数边界、排序规则和验收规则全部写入 artifact。
4. **可拒绝：** 如果候选在 held-out 上变差，必须记录 rejected，而不是包装成正向效果。
5. **可转移：** accepted candidate 能被 Product 侧渲染成 persona prompt、calibration anchor 和报告解释。

论文层面的安全表述：

> S2PC provides an acceptance-gated semantic-structured calibration loop for LLM-driven population simulation, converting segment-level residuals into evidence-linked persona parameter candidates and accepting updates only under held-out distributional alignment.

---

## 4. Product 技术壁垒目标

Product 侧最终需要表达的不是“我们用了更强大模型”，而是：

> 系统能自动发现人群模拟偏差，解释偏差来源，生成可控的人格参数更新，并用独立数据验证更新是否有效。

这比普通 persona simulation 有更强壁垒：

- 普通 demo：写几类人设，让 LLM 生成反应。
- 当前 Product：用 calibration split 给 segment prompt 加锚点。
- S2PC Product：自动把误差转成语义因子和参数化 persona update，并提供 accepted/rejected 证据。

Product 不需要暴露完整搜索细节，但应能展示：

- 哪些 segment 存在明显残差；
- 系统识别了哪些语义因素；
- 哪些人格参数被更新；
- 更新前后 held-out loss 如何变化；
- 该更新是否被 Research gate 接受。

---

## 5. 非目标

S2PC MVP 不做以下事情：

1. 不声称政策因果效应。
2. 不声称中国政策预测能力。
3. 不声称 field validation。
4. 不使用 held-out split 生成候选。
5. 不让 LLM 自由重写完整 prompt 后直接上线。
6. 不把 TextGrad 负结果包装成正向贡献。
7. 不在 Product runtime 中默认启用未通过 runtime effect gate 的 patch。

---

## 6. 输入与输出

### 输入

S2PC MVP 的输入来自 Research 侧现有 artifact：

1. Calibration benchmark artifact
   - schema：`policy-reaction-official-segment-benchmark-v1`
   - split：calibration
   - 必须包含 `segment_metrics`
   - 每个 segment 必须包含 `official_distribution` 和 `predicted_distribution`

2. Initial held-out benchmark artifact
   - split：heldout
   - 提供 baseline loss

3. Candidate held-out benchmark artifact
   - split：heldout
   - 提供 candidate loss

4. Semantic factor catalog
   - MVP 阶段使用本地固定 catalog
   - 后续可以接入文献检索、政策材料检索或客户行业知识库

### 输出

S2PC MVP 输出两个 artifact：

1. `policy-reaction-s2pc-candidate-*.json`
   - 保存 residual、semantic factors、parameter patches、candidate prompt components。

2. `policy-reaction-s2pc-gate-*.json`
   - 保存 initial/best/final loss；
   - 保存 accepted/rejected counts；
   - 保存 source split contract；
   - 保存最终是否 accepted；
   - 可被 `policy_reaction_update_method_gate` 作为一个 method record 引用。

## 7. 核心组件

### 7.1 Residual Miner

职责：从 calibration benchmark 中抽取 segment-policy residual。

定义：

```text
residual(segment, policy) =
  official_distribution(segment, policy)
  - predicted_distribution(segment, policy)
```

输出字段：

```json
{
  "segment": "low_income_food_insecure",
  "policy_id": "food_subsidy_expansion",
  "official_probability": 0.5357,
  "predicted_probability": 0.4120,
  "residual": 0.1237,
  "direction": "under_predicted",
  "magnitude": 0.1237
}
```

Residual Miner 只读取 calibration split。

### 7.2 Semantic Factor Catalog

职责：定义可解释的人群行为因子，以及每个因子可影响的参数。

MVP catalog 包含以下因子：

| factor_id | 中文含义 | 主要影响 |
| --- | --- | --- |
| `food_insecurity_salience` | 食品不安全显著性 | 提高食品补贴类政策支持 |
| `cash_liquidity_preference` | 现金流动性偏好 | 提高现金返还类政策支持 |
| `benefit_immediacy` | 即时收益偏好 | 提高短期可见支持政策 |
| `eligibility_uncertainty` | 资格不确定性 | 降低复杂或资格模糊政策支持 |
| `institutional_trust` | 制度信任 | 提高对政府支持政策的接受度 |
| `household_budget_rigidity` | 家庭预算刚性 | 提高直接生活成本缓解政策支持 |
| `policy_complexity_aversion` | 政策复杂性厌恶 | 降低复杂方案支持 |
| `inflation_stress_sensitivity` | 通胀压力敏感度 | 提高成本缓解政策支持 |

每个 factor 必须定义：

- 适用 segment hints；
- 适用 policy hints；
- 默认方向；
- 可编译参数；
- 参数边界；
- claim boundary。

### 7.3 Semantic Factor Retrieval

职责：根据 residual pattern 从 catalog 中检索语义因子。

MVP 使用规则化检索，不依赖外部向量数据库：

1. 将 residual record 转换成 query features：
   - segment id；
   - policy id；
   - residual direction；
   - residual magnitude；
   - policy attributes。
2. 用 catalog 的 segment hints 和 policy hints 打分。
3. 保留 top-k factors。
4. 输出 factor 与 residual 的映射。

MVP 不让 LLM 直接生成 factor。后续可以加入 LLM critique，但 LLM 输出必须被 catalog schema 校验，不能绕过 catalog。

### 7.4 Semantic-to-Parameter Compiler

职责：把 semantic factor 编译成结构化参数 patch。

示例：

```json
{
  "segment": "low_income_food_insecure",
  "policy_id": "food_subsidy_expansion",
  "factor_id": "food_insecurity_salience",
  "parameter_patch": {
    "food_subsidy_expansion_bias": 0.12,
    "prior_anchor_strength": 0.70,
    "response_temperature": 0.90
  },
  "expected_effect": {
    "food_subsidy_expansion": "increase"
  }
}
```

参数类别：

- `*_bias`：针对 policy 的 segment-level support bias；
- `prior_anchor_strength`：校准分布对 prompt 的约束强度；
- `response_temperature`：输出分布尖锐程度；
- `semantic_salience_weight`：某语义因素在 persona prompt 中的显著性；
- `uncertainty_penalty`：资格不确定或复杂政策惩罚；
- `trust_multiplier`：制度信任对政策接受度的乘数。

编译规则必须是确定性的。同一输入 artifact、同一 catalog、同一 seed 应生成同一候选。

### 7.5 Constrained Parameter Search

职责：在 compiler 生成的参数边界内搜索候选。

MVP 使用 beam search：

1. 每个 residual 生成若干 factor-derived parameter patch。
2. 每个 segment 聚合 top-k patch。
3. 每轮保留 beam width 个候选。
4. 候选评分先用 calibration residual reduction proxy。
5. 只有排名靠前的候选进入 Product run 或 held-out benchmark。

重要边界：

- calibration split 只能用于生成和预排序；
- held-out split 只能用于最终验收；
- 如果没有真实 Product run 生成 candidate benchmark，candidate 只能是 `pending` 或 `diagnostic_only`，不能 accepted。

### 7.6 Prompt/Persona Renderer

职责：把 accepted S2PC 参数渲染成 Product 可消费的 prompt/persona context。

渲染不应暴露原始搜索细节，而应转成可审计的 prompt block：

```text
Accepted S2PC persona calibration:
Segment: low_income_food_insecure
Semantic factors: food_insecurity_salience, benefit_immediacy
Calibration anchor:
- Increase food_subsidy_expansion support sensitivity.
- Keep cash_rebate support bounded by observed calibration prior.
- Preserve strict JSON probability response contract.
```

Product 侧只能消费 accepted S2PC gate；未通过 gate 的 candidate 不进入 runtime prompt。

### 7.7 S2PC Gate

职责：将 S2PC candidate 放入与其他方法一致的验收结构。

Gate 字段：

```json
{
  "schema_version": "policy-reaction-s2pc-gate-v1",
  "artifact_id": "policy-reaction-s2pc-gate-...",
  "overall_status": "accepted",
  "candidate_update_policy": "accept_if_heldout_loss_improves_and_coverage_complete_else_reject",
  "initial_loss": 0.1592,
  "best_loss": 0.0089,
  "final_loss": 0.0089,
  "candidate_accepted_count": 1,
  "candidate_rejected_count": 0,
  "source_split_contract": {
    "residual_mining": "calibration",
    "semantic_factor_retrieval": "calibration",
    "parameter_search": "calibration",
    "candidate_acceptance": "heldout"
  }
}
```

如果 candidate 没有 held-out benchmark，则状态必须是 `diagnostic_only` 或 `pending`。

## 8. 算法流程

```text
Input:
  calibration_benchmark
  initial_heldout_benchmark
  semantic_factor_catalog

Step 1:
  residuals = mine_residuals(calibration_benchmark)

Step 2:
  factor_matches = retrieve_semantic_factors(residuals, semantic_factor_catalog)

Step 3:
  parameter_patches = compile_factors_to_parameters(factor_matches)

Step 4:
  candidate_set = constrained_beam_search(parameter_patches)

Step 5:
  candidate_prompt_components = render_candidate(candidate_set.best)

Step 6:
  Product runs candidate prompt/persona context and exports segment predictions.

Step 7:
  Research evaluates candidate predictions on held-out split.

Step 8:
  S2PC gate accepts candidate only if:
    candidate_loss < initial_loss
    segment_coverage == 1.0
    source_split_contract is valid
```

## 9. 为什么是 1+1>2

结构化参数搜索单独使用的问题：

- 搜索空间稳定，但语义解释弱；
- 参数设计容易显得主观；
- 很难解释为什么某个 segment 应该调整某个 policy bias。

语义检索单独使用的问题：

- 解释力强，但容易生成不可控叙事；
- 如果直接改 prompt，可能出现 TextGrad 式负更新；
- 缺少可复现的搜索边界。

S2PC 的融合增益：

1. 语义检索缩小参数搜索空间，避免盲目搜索。
2. 结构化参数约束语义更新，避免自由文本漂移。
3. Residual 让语义因素有数据来源，而不是凭空解释。
4. Held-out gate 让语义解释必须接受独立验证。
5. Product 可以同时展示“为什么调”和“调完是否有效”。

8. 将 S2PC method record 接入 `policy_reaction_update_method_gate.py`。

MVP 不要求 Product runtime 立即消费 S2PC artifact。只有当 S2PC candidate 在 held-out gate accepted 后，才进入 Product runtime 渲染。

## 11. 与现有方法的关系

| 方法 | 当前地位 | S2PC 中的角色 |
| --- | --- | --- |
| `calibration_split_prompting` | 当前 best baseline | 必须继续作为强 baseline |
| `policy_reaction_residual_parameter` | patch gate 可生成，但 runtime stability rejected | 可作为 S2PC 的 residual 输入和对照 |
| TextGrad / DeepSeek v4-pro | diagnostic-only | 可作为 semantic factor proposal 的候选来源，但不能直接 accepted |
| S2PC | 新方法 | 目标是在解释性和 held-out loss 上同时优于简单 calibration split 或至少优于 DeepSeek v4-flash calibration split |

## 12. 测试方案

### 单元测试

1. Residual Miner
   - 正确计算 residual；
   - 拒绝非 calibration split；
   - 拒绝缺失 distribution 的 artifact。

2. Semantic Factor Catalog
   - catalog strict JSON；
   - 每个 factor 都有参数边界；
   - 每个 factor 都有 claim boundary。

3. Semantic Factor Retrieval
   - 同一 residual 输入返回稳定 top-k；
   - policy hint 和 segment hint 都进入评分；
   - magnitude 低于阈值的 residual 不生成 factor。

4. Semantic-to-Parameter Compiler
   - 参数方向与 residual direction 一致；
   - 参数值不超过 factor 边界；
   - 同一输入 deterministic。

5. Constrained Parameter Search
   - beam width 生效；
   - 搜索结果按 proxy score 排序；
   - 输出候选保留 provenance。

6. S2PC Gate
   - held-out loss 改善时 accepted；
   - held-out loss 变差时 rejected；
   - coverage 不完整时 rejected；
   - 没有 held-out benchmark 时 diagnostic-only。

### 集成测试

1. 用当前 calibration benchmark 生成 S2PC candidate artifact。
2. 用 fixture candidate held-out benchmark 生成 accepted/rejected gate。
3. 将 S2PC gate 转成 update-method record。
4. 运行 update-method gate，确认 S2PC 与 TextGrad、calibration split、runtime patch 能同表比较。

### 验证命令

```bash
.venv/bin/python -m pytest tests/test_policy_reaction_s2pc.py -q
.venv/bin/python -m pytest -q
git diff --check
```

## 13. 验收标准

MVP 完成必须满足：

1. 新增 S2PC candidate artifact，严格 JSON。
2. 新增 S2PC gate artifact，严格 JSON。
3. S2PC gate 明确记录 initial/best/final loss。
4. S2PC gate 明确记录 accepted/rejected/diagnostic。
5. 所有 candidate generation 字段都指向 calibration split。
6. 所有 acceptance 字段都指向 held-out split。
7. S2PC method record 能进入 `policy-reaction-update-method-gate-current-001`。
8. 如果没有真实 held-out candidate benchmark，最终状态不能是 accepted。
9. 全量 Research 测试通过。
10. 文档和 artifact 不包含 provider key。

研究性验收目标分两级：

- **MVP 合格：** 能生成可解释、可审计、可拒绝的 S2PC candidate，并进入 method gate。
- **论文贡献候选：** 至少在 DeepSeek v4-flash 或其他非最佳 baseline 上稳定优于简单 calibration split，并在多个 seed/scale 上保持 accepted。

## 14. 风险与控制

### 风险 1：语义因子 catalog 主观

控制方式：

- 每个 factor 必须有 segment/policy 适用边界；
- 每个 factor 必须有参数边界；
- 每个 factor 必须能追溯到 residual；
- 论文中将 catalog 描述为 method design，不把它说成经验真理。

### 风险 2：参数搜索过拟合 calibration split

控制方式：

- calibration split 只用于生成和预排序；
- held-out split 只用于验收；
- 后续必须增加 seed/scale repeat；
- Product 不展示未通过 held-out gate 的 candidate。

### 风险 3：解释性强但指标不提升

控制方式：

- 记录 rejected；
- 不进入 Product runtime；
- 在 paper 中作为 negative result 和方法边界讨论。

### 风险 4：LLM 语义建议不稳定

控制方式：

- MVP 不依赖 LLM 自由生成 factor；
- 后续 LLM proposal 必须映射到 catalog schema；
- 无法映射的建议只能 diagnostic-only。

## 15. Product 转移路径

Product 侧不直接实现搜索。Product 只消费 Research accepted artifact。

转移步骤：

1. Research 生成 accepted S2PC gate。
2. Product runner 校验：
   - schema version；
   - overall_status；
   - source split contract；
   - runtime effect gate。
3. Product 将 accepted S2PC 参数渲染进 persona prompt。
4. Product manifest 记录：
   - S2PC gate id；
   - accepted factor ids；
   - accepted parameter patch count；
   - claim boundaries。
5. Product report 展示：
   - residual before；
   - semantic factors；
   - parameter update；
   - held-out improvement；
   - claim boundary。

这个路径确保 Product 的说服力来自 Research 证据，而不是 demo 叙事。

## 17. 结论

S2PC 的价值在于把 TextGrad 式自由 prompt 更新改造成可审计的社会计算校准方法：

- 用 residual 提供数据起点；
- 用 semantic factor 提供解释；
- 用 structured parameter 提供可控更新；
- 用 constrained search 提供自动化；
- 用 held-out gate 提供证据边界；
- 用 accepted artifact 服务 Product 技术壁垒。

这条线可以成为 Research 的方法创新点，也可以成为 Product 区别于普通 LLM persona demo 的核心护城河。
